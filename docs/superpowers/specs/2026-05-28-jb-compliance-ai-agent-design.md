# JB 준법 코파일럿 — 소비자 오인 방지 중심 준법자문 AI Agent (설계 문서)

- **공모전**: JB금융그룹 Fin:AI Challenge (DACON)
- **주제**: 지정주제2 — Compliance AI / 준법자문가 AI Agent 서비스 개발
- **작성일**: 2026-05-28
- **참가**: 솔로 (개발은 Claude 수행)
- **하드 마감**: 2026-06-12 / 개발 목표: ~1주 (~2026-06-04)
- **제약**: 무료/오픈소스 모델만 사용 (상용 API 금지), 온프레미스 구동

---

## 1. 한 줄 요약

대고객 금융 콘텐츠를 **작성 시점부터** 심의하고, **소비자가 어떻게 오해하는지를 시뮬레이션**해 위험을 드러내며, **규제 근거와 함께 수정안**을 제시하고, 준법관리자는 **검토·승인만** 하는 **온프레미스 AI Agent**.

---

## 2. 문제 정의 (은행 제시 내용 기반)

**현행 분석 (은행 제시)**
- 대고객 콘텐츠 전반을 준법관리자가 대부분 **수작업** 심의.
- 외국어(다국어) 콘텐츠도 동일 인력이 동일 방식으로 수행.
- 심의 지연 → 콘텐츠 배포 지연 → **마케팅 적시성** 손실.
- 심의 **품질 편차** 및 휴먼에러 가능성.

**한계점 (은행 제시)**
- 준법 심의가 **병목** → 콘텐츠 적시성·확장성 제한.
- 다국어·다채널 확장 시 심의 리소스 **선형 증가**.
- 규제 변경의 **즉시 반영 어려움** → 대응 리스크 지속.

**문제의 본질**: 금융 광고 규제(금융소비자보호법·표시광고법 등)의 핵심 목적은 "규칙 위반 적발"이 아니라 **소비자가 오인·기만당하는 것을 방지**하는 것이다. 기존 도구·수작업 심의는 "표현 매칭" 중심이라 *실제 소비자가 어떻게 오해하는가*를 직접 다루지 못한다.

---

## 3. 솔루션 컨셉 & 차별점

은행이 제시한 **기대 방향성·기능 예시를 골격으로 그대로 구현**하고, 그 위에 차별점 3가지(+α)를 얹는다.

**+α 1 — 소비자 오인 시뮬레이터 (헤드라인 / 메인)**
고령자·금융초보·외국인 거주자 페르소나가 광고를 *1인칭으로 읽고 오해*하는 과정을 재현 → 위험 오해 + 유발 문구 + 금소법 '오인 방지' 조항을 연결. 경쟁 제품(Saifr, PerformLine 등)도 하지 않는 독보 지점이며, 규제의 본질을 정면으로 공략.

**+α 2 — 작성 시점 실시간 예방 (경량)**
"사후 배치 심의"가 아니라 작성 중(멈춤/붙여넣기 시) 자동 검사 → 현행 문제(심의 지연·적시성)를 근본 해결.

**+α 3 — 온프레미스 / 오픈모델 (상시 적용)**
은행은 준법 심의 데이터를 외부 SaaS에 전송 불가(기밀·규제). 오픈모델 기반 온프레미스 = **은행이 실제 도입 가능한 유일한 형태**. "외부 API 미사용"이 한계가 아니라 **정당성**이 된다.

**경쟁 환경 (조사 결과)**: 해외에 금융 마케팅 준법 AI(Saifr/Fidelity, PerformLine 등)가 존재하나 ① 영어·미국/EU 규제 기반, ② 업로드형 사후 검토/모니터링, ③ SaaS. **한국어·한국 규제 기반 + 작성 시점 + 소비자 오인 시뮬레이션 + 온프레미스** 형태의 공개 제품은 없음.

## 4. 은행 요구사항 ↔ 기능 매핑

| 은행이 제시한 내용 | 본 서비스의 대응 기능 |
|---|---|
| (기대①) AI가 최신 규제+내부기준 자동 추적 | 규제 지식베이스 + 인제스트 파이프라인(문서 버전관리), 내부 심의기준 등록 |
| (기대②) 초안의 위반가능성·표현리스크·수정제안 자동 도출 | 심의 엔진: 위반 탐지 → 리스크 등급 → 인라인 수정안 |
| (기대③) 준법관리자는 검토·승인 역할로 전환 | Human-in-the-loop 승인 워크스페이스 (AI 1차심의 → 사람 승인/반려) |
| (기대④) 승인 결과를 마케팅/제작 프로세스와 자동 연계 | 승인 → 배포가능 상태 전환 + 내보내기 + 감사로그(audit trail) |
| (기능①) 규제 문서 검색·참조로 근거 제공 | RAG: 모든 지적에 규제 조항 출처 인용 |
| (기능②) 다국어 콘텐츠 이해 + 리스크 분류 | 다국어 심의(외국인 고객 콘텐츠) + 번역 누락/왜곡 검증 |
| (기능③) 콘텐츠 유형별 기준 구조화 + 룰+LLM 결합 | 유형 분류기 + 룰엔진(결정적) + LLM(뉘앙스) 하이브리드 |
| (현행/한계) 휴먼에러·품질편차·병목·적시성 | 일관된 자동 1차심의로 편차↓ · 적시성↑ |

## 5. 시스템 아키텍처

```
┌────────────────────────── Frontend (Next.js + React + Tailwind) ──────────────────────────┐
│  심의 워크스페이스 · 오인 시뮬레이터 패널 · 준법관리자 승인 뷰 · 다국어 비교 뷰              │
└───────────────────────────────────────────────┬───────────────────────────────────────────┘
                                                 │ REST (JSON)
┌───────────────────────────────────── Backend (FastAPI) ───────────────────────────────────┐
│  Orchestrator (명시적 파이썬 파이프라인)                                                     │
│   ├─ 콘텐츠 유형 분류기                                                                      │
│   ├─ Retriever (RAG)  ──→  Vector DB (Chroma) ←── 규제 코퍼스 인제스트/인덱싱               │
│   ├─ 룰엔진 (YAML 규칙 + 정규식/키워드)                                                      │
│   ├─ LLM 심의기  ─────┐                                                                      │
│   ├─ 오인 시뮬레이터 ─┼─→  LLM 서버 (Ollama: Qwen2.5-7B-Instruct)                            │
│   ├─ 리포트 종합기    ┘     임베딩 (BGE-m3)                                                  │
│   ├─ 다국어 생성+재심의                                                                      │
│   └─ 승인/감사로그 (SQLite)                                                                  │
└────────────────────────────────────────────────────────────────────────────────────────────┘
```

전부 로컬 구동(온프레미스). 무거운 에이전트 프레임워크(LangGraph 등) 대신 **명시적 파이프라인**으로 디버깅·설명가능성·데모 투명성 확보.

```
콘텐츠 초안 입력 (한국어/외국어, 유형 자동/수동)
 ① 콘텐츠 유형 분류      → 예금·투자·대출 (유형별 심의기준 라우팅)
 ② 규제 근거 검색(RAG)   → BGE-m3 임베딩 → Chroma top-k 조항          ┐ (②③ 병렬)
 ③ 룰엔진(결정적)        → 금지표현·필수고지 누락·금리표기 규칙        ┘
 ④ LLM 심의(뉘앙스)      → ②근거+③결과로 위반가능성·표현리스크·수정안(출처 인용)  ┐ (④⑤ 병렬)
 ⑤ ★오인 시뮬레이터      → 페르소나별 오해 생성 + 위험 오해 + 유발문구 + 조항      ┘
 ⑥ 리포트 종합          → 위반·등급·수정안·오인 시나리오·출처를 한 화면에
 ⑦ 준법관리자 검토·승인  → 승인/반려/코멘트 (Human-in-the-loop)
 ⑧ 승인 후              → 배포가능 + 감사로그 + (다국어면) 타깃언어 생성 + 재심의
```

## 7. 핵심 엔진 — 소비자 오인 시뮬레이터 (헤드라인)

**목표**: 광고 초안 → 취약·다양한 소비자가 어떻게 오해하는지 1인칭 재현 → 위험 오해를 유발 문구·조항과 연결.

**동작**
1. 페르소나별 병렬 LLM 추론. 각 페르소나 프로필(금융이해도·관심사·편향) + 광고 텍스트 주입.
2. 프롬프트: "당신은 이 사람입니다. 이 광고를 읽고 ①무엇을 이해했는지 ②어떤 행동을 할지 ③무엇을 기대/오해했는지를 1인칭으로."
3. 오해 판정기(judge): 페르소나의 '이해'를 ⓐ상품의 실제 조건(구조화 입력) + ⓑ규제상 '암시 금지' 항목(원금보장·확정수익 등)과 대조 → 불일치를 **오해**로 플래그(위험도 부여), 유발 문구를 LLM으로 역추적.
4. 출력: 페르소나별 [이해 요약 / 위험 오해 🔴 + 유발 문구 하이라이트 + 금소법 조항 / 안전 🟢] + 종합 오인 리스크 점수.

**페르소나 (MVP 3종)**
- 👵 72세 은퇴자 — 금융이해도 낮음·안정지향 → "원금 보장" 오해 취약.
- 🧑 20대 금융초보 — 용어 약함·단기수익 기대 → "확정 수익률" 오해.
- 🌏 외국인 거주자 — 한국어 비모국어·제도 이해 낮음(전북은행 특성) → 조건·제도 오해 + 다국어 검증 연결.

**준법 골격과의 연결**: 오인 시뮬레이터 결과는 ④ LLM 심의의 '표현 리스크' 판정을 **강화하는 근거**가 된다 (예: "이 문구는 72세 고객에게 원금보장으로 오인됨 → §22 위반 소지"). 차별 기능이 본 심의의 **설명가능성**을 높이는 구조.

## 8. 데이터 설계

### 8.1 규제 지식베이스 (RAG 코퍼스)
공개 자료 기반, MVP는 예금·투자·대출 핵심 조항 큐레이션.
- 금융소비자보호법(§22 광고 준수사항) + 시행령
- 금융상품 등의 표시·광고에 관한 심사지침
- 표시·광고의 공정화에 관한 법률
- 협회 광고심의 가이드라인 (은행연합회 / 금융투자협회)
- 자본시장법 투자광고 조항
- (가상) JB 내부 심의 체크리스트 — "내부 기준 자동 추적"(기대①) 시연용
- 메타데이터: 법령명·조항·시행일·출처URL → 출처 인용 + 버전관리(규제 변경 추적 토대)

### 8.2 룰엔진 정의 (결정적 체크, YAML)
규제에서 파생. 각 룰: `id / 적용유형 / 패턴(정규식·키워드) / 위반근거(규제 매핑) / 심각도 / 수정 가이드`.
- 금지 단정·보장 표현: `원금 보장`·`확정 수익`·`최고`·`100%`·`무조건` → 손실보전·이익보장 오인 금지
- 필수 고지 누락: 투자=`원금손실 가능`, 예금=`예금자보호 한도`, 대출=`연체이자율/중도상환수수료`
- 금리/이율 표기: `연 이율` 명시·세전세후·우대조건
- 과장 최상급: `업계 1위`·`최저금리` 등 객관적 근거 없는 표현

### 8.3 콘텐츠 유형 분류 (MVP 3종)
예금·적금 / 대출 / 투자(펀드) — 유형별 필수고지·금지표현 체크리스트(기능③).

### 8.4 테스트 픽스처 (= 데모 입력 = 테스트셋)
위반이 이미 심어진 샘플 광고 초안(한국어 + 다국어 1). 예: *"확정 연 5%, 원금 보장 예금!"* → 기대 위반: 단정표현·원금보장 오인·필수고지 누락. TDD 정답셋으로 사용(알려진 입력 → 기대 플래그).

## 9. 기술 스택 (전부 오픈/무료)

| 레이어 | 선택 | 비고 |
|---|---|---|
| LLM | Qwen2.5-7B-Instruct (Q4) | 한국어+다국어 강함, GPU 8–12GB 쾌적, 14B-Q4 선택 가능 |

| 백엔드 | FastAPI | 비동기 오케스트레이션 |
| 룰엔진 | YAML + 정규식/키워드 | 결정적·설명가능 |
| 저장소 | SQLite | 콘텐츠·심의이력·감사로그 |
| 프론트 | Next.js + React + Tailwind | 데모 폴리시 |

### 10.3 다국어 (승인 후)
영어 버전 자동 생성 → 재심의로 `guaranteed` 같은 오역·필수고지 누락 검출. KO 승인본 ↔ EN 생성본 비교 뷰.

1. *"확정 연 5%! 원금 보장 안심예금"* 붙여넣기 → 멈춤 → 자동검사

## 12. MVP 범위 vs Stretch

**MVP (필수)**: F1–F11 + 규제 코퍼스(3유형) + 페르소나 3종 + 워크스페이스/승인뷰/감사로그 + 테스트 픽스처.

## 15. 에러 처리 & 엣지 케이스

- LLM 출력 파싱 실패 → 구조화 출력(JSON 스키마) 강제 + 재시도, 실패 시 룰엔진 결과만으로 폴백.
- RAG 무관 조항 검색 → 점수 임계값 + top-k 제한, 근거 부족 시 "근거 불충분" 표기(허위 인용 방지).
- 빈/초단문 입력 → 검사 스킵 + 안내.
- 다국어 번역 오역 → 재심의로 검출(필수고지 보존 여부 대조).
- LLM 환각으로 없는 조항 인용 → 인용은 **검색된 코퍼스 내 조항으로 제한**(LLM이 조항번호를 창작하지 못하도록 retrieved 컨텍스트의 메타데이터만 인용).
