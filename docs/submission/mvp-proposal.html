1Summary

JB 준법 코파일럿은 대고객 금융 광고를 작성하는 그 순간, 고령자·금융초보·외국인 소비자가 그 문구를 어떻게 오해하는지 1인칭으로 재현해 보여주고, 규제 근거와 함께 수정안을 제시하며, 위반 초안을 준법 통과까지 자율 개선하는 온프레미스 AI Agent다. 준법관리자는 전수 초벌심의 대신 검토·승인에 집중한다.

2문제 정의 (Problem Definition)

2.1 현행 업무의 병목·리스크

  • 사후·수작업 심의의 한계: 은행 광고는 게시 전 준법감시인 사전심의가 의무(금소법 §22, 은행연합회 광고심의규정)지만, 현실은 "초안 → 수일 대기 → 수작업 심의 → 반려·재작성 반복"의 왕복으로 적시성을 잃는다.
  • "합법이지만 오해를 부르는" 사각지대: 규정 문구만 점검하면 통과해도, 고령자·외국인이 실제로 오해하는 표현은 걸러지지 않아 불완전판매·민원으로 이어진다. 규제의 본질은 "오인 방지"인데 기존 심의는 "표현 점검"에 머문다.
  • 제재 리스크 직결: 금소법 §22 위반 광고는 과징금·기관제재 대상. 광고 1건이 분쟁의 시작점이 될 수 있다.

2.3 JB 관점의 중요도

전북·광주은행은 지역 밀착 마케팅 빈도가 높고, 전북 지역 외국인 거주자 고객 비중이 늘어 다국어 콘텐츠 수요가 커지는 반면 심의 인력은 한정적이다. 입력(초안)·출력(심의·승인)·사용자·프로세스 접점이 명확해 그대로 내부 시스템 과제로 전환 가능하며, 본 MVP가 그 실현성을 동작으로 증명한다.

3제안 솔루션 개요 (Solution Overview)

작성 시점에 작동하는 온프레미스 준법 AI Agent. 명시적 파이썬 파이프라인이 ①콘텐츠 유형 분류 → ②규제 근거 검색(RAG) → ③룰엔진(결정적) → ④LLM 심의(뉘앙스) → ⑤소비자 오인 시뮬레이터 → ⑥리포트 종합 → ⑦⑧준법관리자 승인·감사로그로 흐른다.

┌─ Frontend (정적 SPA · FastAPI 서빙) ─ 헤드라인 오인 시뮬레이터 · 에디터 · 승인 뷰 ─┐
│                                   │ REST(JSON)                                     │
├─ Backend (FastAPI · 명시적 파이프라인) ───────────────────────────────────────────┤
│ ①유형분류 → (②RAG ∥ ③룰엔진) → (④LLM심의 ∥ ⑤오인 시뮬레이터) → ⑥리포트 → ⑦⑧승인·감사 │
│ ★오토파일럿: 통과까지 자율 개선  ·  내규/규제추적/배포/상품DB 연동                  │
│ LLM: Ollama(Qwen2.5-7B) — 미연결 시 룰 기반 결정적 폴백 (오프라인에서도 전 기능)     │
└──────────────────────────────────────────────────────────────────────────────────┘
신뢰 모델 — 룰엔진이 권위, LLM은 보조판정의 뼈대는 결정적 룰엔진(YAML 공개)과 RAG 인용이고, LLM은 뉘앙스·역할극을 담당하되 인용은 검색된 조항으로 제한되고 출력은 룰엔진이 검증한다. 환각이 등급을 좌우할 수 없는 구조 — 은행이 실제로 받아들일 수 있는 유일한 형태.

4주요 기능 정의 (Key Features)

★ 소비자 오인 시뮬레이터 헤드라인

👵72세·🧑20대·🌏외국인 페르소나가 광고를 1인칭으로 읽고 오해 → 위험 오해 + 유발 문구 + 규제 조항 연결. 화면을 열면 가장 먼저 보인다.

준법 오토파일럿 Agent

위반 초안을 심의→고쳐쓰기→재심의로 🟢통과까지 자율 반복. 합격은 룰엔진이 검증, 게시는 사람 승인.

작성시점 실시간 예방

타이핑을 멈추면 즉시 룰 검사(0.03초) → LLM 심의 보강. 위험 문구 자동 밑줄.

규제 근거 RAG + 인용

모든 지적에 금소법·심사지침 등 조항 출처 인용. 근거 부족 시 "근거 불충분" 표기.

내부 심의기준(내규)

준법부가 평문으로 내규 입력 → AI가 룰로 변환 → 법규와 함께 적용 (비개발자·〔내규〕).

규제 자동 추적

규제 피드 변경 감지 → 영향 콘텐츠 분석 → 룰 자동 제안(사람 승인). 운영: 법제처 OpenAPI.

승인 + 감사로그

준법관리자 승인/반려. 누가·언제·어떤 근거로를 영구 기록 (내부통제·검사 대응).

마케팅 배포 Last-Mile

승인 콘텐츠 → 채널(푸시·SMS·이메일·SNS·CRM) 자동 연계. 미승인 차단. 운영: FCM/SNS/알림톡.

다국어 + 신뢰도 방어

승인본 외국어 생성·재심의(직역 오역 검출). 비-한/영은 신뢰도 플래그+사람검토.

그 외: 상품 DB 연동(상품코드→조건 자동 주입, 코어뱅킹 슬롯) · 수정안 원클릭 적용 · 콘텐츠 유형 분류 라우팅. 전체 F1–F16 구현, 자동 테스트 53건 통과. 상세는 함께 제출하는 기능명세서 참조.

5.1 데이터 출처 · 정직성

  • 실제 규제 기반: 금소법 §22, 표시광고법 §3, 자본시장법 §57, 표시·광고 심사지침, 은행연합회·금융투자협회 규정 — 국가법령정보센터(법령은 저작권법 §7 비보호 저작물). 내용은 실제 규제의 취지를 정확히 반영.
  • 시연용(가상) 데이터 명시: MVP 코퍼스는 요약·재구성본(12조항)이며, JB 내부기준·규제 피드·상품 카탈로그는 가상 데이터(시연용)다. 운영 시 원문 전문 수록·법무 검증 + 실제 내규/법제처 OpenAPI/코어뱅킹 API로 교체하는 어댑터 슬롯으로 설계.
  • 상용 API 미사용: 대회 제약(무료/오픈소스) 및 온프레미스 원칙 준수. 외부 전송 0.

5.2 환각·책임 방어 (설계에 내장)

인용 화이트리스트(검색 조항으로 제한)·quote 원문 검증·룰엔진 권위·휴먼인더루프·감사로그·비영어권 신뢰도 플래그. 모델 교체는 환경변수 1개로 오픈모델 발전을 즉시 수용.

6사용자 시나리오 / 유즈케이스 (User Scenario)

준법감시부·마케팅부의 하루 — 작성 시점 예방부터 배포 연계까지 한 흐름:

  1. [상시] 규제 자동 추적 — 새벽에 규제 피드(운영: 법제처 OpenAPI)에서 개정 감지 → 영향 콘텐츠 분석 → 룰 변경 제안. 준법감시인이 검토·승인으로 룰셋 최신화.
  2. [작성] 실시간 예방 — 담당자가 초안 작성. 상품코드 선택 시 코어뱅킹에서 조건 자동 주입. 멈추면 즉시 룰 검사 → LLM 보강. 헤드라인에 "👵 72세가 '원금 보장이니 안전하네'로 오해 🔴"가 떠 작성 중에 위험 인지.
  3. [자가 개선] — [🚀 자동 개선]으로 통과까지 자율 수정(미수렴 시 사람 에스컬레이션). 법규 + JB 내규 동시 적용.
  4. [검토·승인] — 준법감시인이 AI 1차심의·페르소나 오해·규제 인용을 근거로 승인/반려. 결정이 감사로그에 영구 기록.
  5. [배포 연계] — 승인 즉시 채널(푸시·SMS·이메일·SNS·CRM)로 자동 발송(미승인 차단). "AI 자동개선 → 사람 승인 → 시스템 배포"가 분리 기록되어 금감원 검사 대응.
  6. [다국어] — 전북 외국인 고객용 외국어 버전을 승인본에서 자동 생성·재심의. 비-한/영은 신뢰도 플래그+사람검토.
전 과정 온프레미스 + 휴먼인더루프모든 자동화(규제 룰·자가 개선·배포)는 사람의 승인을 거친다. AI는 1차심의·권고만, 배포 결정 권한은 항상 준법관리자에게 있어 책임 경계가 명확하다.

7기대 효과 및 향후 확장성 (Expected Impact)

7.1 기대 효과

축효과
업무효율준법관리자의 초벌 심의(검출·근거 탐색·수정안 작성)를 자동화 → 건당 검토를 "읽고 판단"으로 축소. 다국어 심의의 한계비용 ≈ 0.

고객가치취약계층(고령·초보·외국인) 관점의 오해를 사전 제거 → 명확·공정한 정보, 신뢰 제고.
수익기회심의 리드타임 단축 → 마케팅 적시성 회복(시즌·금리 이벤트), 콘텐츠 생산량 확대.

7.3 확장 축

  • 계열사: 룰·코퍼스의 types 라우팅 구조가 그대로 확장 단위 — 은행 광고 → 캐피탈(대출성) → 자산운용(투자광고).
