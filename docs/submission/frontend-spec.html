1개요 · 설계 원칙

프론트엔드는 준법 심의를 "문서 작성 도구"의 경험으로 제공한다. Grammarly·린터처럼 작성 중 실시간으로 위험 표현에 물결 밑줄을 긋고, 화면 최상단(헤드라인)에 「👥 소비자는 이렇게 오해합니다」를 띄워 작성자가 배포 전에 위험을 체감하게 한다. 별도 빌드·번들러 없이 단일 HTML로 1-커맨드 구동되며, 백엔드 폴백 모드와 맞물려 오프라인에서도 전 기능 시연이 가능하다.

2.2 심의 워크스페이스 — 검토 패널 (우측)

섹션표시 내용
① 준법 점수 게이지n/100 + 색상 게이지(🟢통과·🟡주의·🔴위험). 점수 변동은 0.55초 ease 트랜지션으로 애니메이션
① 다국어 신뢰도 배너비-한/영 언어일 때 "다국어 베타 — 룰엔진이 판정 권위, AI는 참고용, 사람검토 권장" 경고 배너
② 👥 소비자 오인 (★헤드라인)페르소나 5종(👵 72세 은퇴자·🧑 20대 초보·🌏 외국인 + 🌾전북 농업인·🏪전북 소상공인, 「전북 특화」 배지)이 1인칭으로 오해 → 🔴위험 오해 + ⟵ 유발 문구(클릭 시 본문 이동) + 금소법 조항. "참고지표 — 법적 판정 아님" 명시

2.4 준법관리자 검토 뷰

  • 검토 대기열: 제출 목록(제목·제출자·유형·등급 요약). 사이드바 배지에 pending 건수 표시.
  • 제출 상세: 본문 + AI 리포트(점수·페르소나 오해·규제 인용) + 승인 / 반려 / 코멘트 버튼.
  • 승인 후 액션: [📤 마케팅 채널 자동 배포](F15, 미승인 차단) · [🌏 영문 버전 생성+재심의](F10).
  • 다국어 재심의 결과 배너(F10): 번역본마다 재심의 등급 표시 — 통과 시 🟢"외국어 채널 후속 배포 가능", 불통과 시 🔴"재심의 불통과 — 마케터·준법감시인 확인 필요, 외국어 채널 배포 보류".
  • 감사 로그: 제출·자동개선·승인·배포·번역·재심의 알림이 누가/언제/근거로 시간순 표시(F9).

2.5 모달 — 오토파일럿 자율 개선 애니메이션

백엔드 /api/autopilot가 반환한 회차 trace를 받아 한 회차씩 순차 재생한다: 점수 게이지가 회차마다 상승(animateScore), 회차별 변경내역(before→after·근거 조항)이 스텝 리스트로 쌓이고, 종료 시 결과 헤더(passed/plateau/regressed/사람 에스컬레이션)와 [검토로 전송] 노출. "심의 도구"가 아니라 "통과시켜 주는 에이전트"임을 시각적으로 증명.

종료 후 부가 표시(피드백 반영):

  • 수정 전/후 비교 뷰(diff) F12: 초안→최종을 어절 단위로 <del>(삭제·적색 취소선)·<ins>(추가·녹색)로 인라인 렌더 + 「− N / + N 어절」 요약. AI 수정 내역을 한눈에(감사 추적).
  • 가독성 참고지표 F12: 점수·라벨(쉬움/보통/어려움)·게이지 + 평균 문장 길이·전문용어. "준법 점수와 별개 · 마케팅 효과 아님"을 라벨에 명시해 과대표현·오해 방지.

3.1 실시간 검사 — 2단계 디바운스 (F11)

작성 중에는 결정적 룰 검사·점수만 즉시 갱신하고, LLM이 필요한 소비자 반응 시뮬레이션은 충분히 멈췄을 때만 발화한다. 매 키 입력마다 LLM을 호출하면 로컬 7B 모델 큐가 적체돼 타이핑이 밀리기 때문이다.

3.2 경합 제어 — 최신 요청만 반영

let reviewSeq = 0, reviewAbort = null;
async function runReview(quick){
  if(reviewAbort) reviewAbort.abort();      // ① 진행 중 fetch 중단(밀림·stale 방지)
  const ac = new AbortController(); reviewAbort = ac;
  const seq = ++reviewSeq;                   // ② 요청 시퀀스 발급
  const rep = await api("/api/review", { signal: ac.signal, ... });
  if(seq !== reviewSeq) return;              // ③ 더 최신 요청이 있으면 이 응답 폐기
  state.report = rep; render(rep);
}

근거: 타이핑 중 여러 심의가 겹치면(특히 느린 LLM 응답) 오래된 결과가 최신 화면을 덮어쓸 수 있다. AbortController로 직전 요청을 끊고, 시퀀스 번호로 "최신 요청의 응답만" 렌더링해 화면 일관성을 보장한다. AbortError는 정상(새 입력)으로 무시.

3.3 물결 밑줄 오버레이 합성

  • 백엔드 리포트의 highlights(룰 span + LLM quote 위치 + 오인 유발 문구, start 정렬)를 받아, 겹치지 않는 구간만 골라(h.start >= last) <mark>로 감싼다 — 중첩 밑줄로 인한 깨짐 방지.
  • 오버레이 텍스트는 항상 현재 에디터 본문과 동일(state.report.text !== text면 일반 텍스트로 폴백) → 입력과 밑줄 위치 어긋남 방지.
  • HTML 이스케이프(esc) 후 삽입 — 사용자 입력의 <,& 등 XSS·렌더깨짐 차단.

6접근성 · 견고성

  • 키보드: 에디터 포커스 유지, 셀렉트·버튼 표준 포커스 링(--ring). 모달은 오버레이 클릭/닫기 버튼.
  • 입력 안전: 모든 사용자 텍스트는 esc() 이스케이프 후 DOM 삽입 — 밑줄 오버레이·말풍선·리스트 전부.
  • 오류 처리: api()가 비정상 응답을 throw → 라이브 표시등에 오류 라벨. 네트워크 중단(AbortError)은 정상 흐름으로 무시.
  • 폴백 인지: 상태바 엔진 배지가 LLM/폴백을 항상 표시 → 시연 환경에서 현재 동작 모드를 투명하게 안내.
  • 빈 상태: 모든 패널에 안내형 empty 메시지 — 첫 사용자도 다음 행동을 알 수 있음.
